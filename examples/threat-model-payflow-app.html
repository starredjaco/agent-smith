Shostack 4-Question Session Summary

1. What are we working on? A SaaS payment platform processing card transactions, storing PII, and exposing a REST API to merchant integrations. In PCI DSS Level 1 scope.
2. What can go wrong? Card data exfiltration via SQL injection; account takeover via JWT bypass; admin portal authorization bypass; webhook replay attacks; logic manipulation of payment amounts and refund flows.
3. What are we going to do about it? Immediate: fix JWT validation, parameterized queries, admin IDOR. Short-term: WAF rules, SAST in CI, rate limiting. Long-term: PCI DSS gap assessment, event sourcing for financial state.
4. Did we do a good job? ✅ Coverage · ✅ Depth (logic flaws + chained attacks) · ✅ Actionability · ✅ Business alignment
1

Objectives & Scope

Business Objectives

  • Process card-not-present transactions on behalf of merchant customers
  • Maintain PCI DSS Level 1 compliance (>6M transactions/year)
  • Ensure 99.9% API uptime SLA for enterprise merchants
  • Protect cardholder data and merchant financial data from breach

Assets to Protect

  • Cardholder Data (CHD) — PAN, CVV (transit only), expiry. Highest sensitivity.
  • Merchant API Keys — Full payment authorization capability if leaked.
  • Transaction Records — Financial and PII data with regulatory retention requirements.
  • Admin Portal Access — Full platform control; all merchant data visible.

Compliance Constraints

  • PCI DSS v4.0 — Mandatory for card processing
  • GDPR — EU merchant and cardholder data
  • SOC 2 Type II — Enterprise customer requirement

Business Logic Flaws

🧠 Activity 3: Logic Flaws Identification

These vulnerabilities are not found by standard scanners or STRIDE alone — they require understanding the intended business behavior and thinking like an attacker targeting it.

BL-01 — Negative Amount Injection

FlowPOST /v1/charges — amount field
FlawNo server-side validation that amount > 0. Sending amount: -500 causes a credit (refund) without a corresponding charge.
WhoAny authenticated merchant with a valid API key
ImpactFinancial loss, triggers regulatory audit

BL-02 — Race Condition in Refund Flow (TOCTOU)

FlowPOST /v1/charges/{id}/refund
FlawRefund eligibility checked (status == completed), then refund processed — no atomic lock. Concurrent requests pass the check simultaneously and issue multiple refunds.
WhoAny merchant integration capable of concurrent requests (trivial with async HTTP)
ImpactDuplicate refunds; financial loss proportional to transaction volume

BL-03 — Webhook Replay Attack

FlowWorker → Merchant webhook delivery
FlawWebhook includes event_id but no timestamp. A captured webhook can be replayed indefinitely. Combined with T-06 (no HMAC verification), enables repeated order fulfillment.
WhoAttacker with access to a logged/captured webhook payload
ImpactRepeated order fulfillment, inventory manipulation

BL-04 — Step-Skip in 3DS Authentication Flow

Flowcharges → authenticate → capture (3-step)
Flawcapture endpoint does not verify that authenticate was completed. Directly calling capture after charge creation succeeds, bypassing 3DS.
WhoAny merchant API key holder
ImpactChargeback liability remains with platform, not card network — financial and compliance exposure

BL-05 — Privilege Context Collapse in Multi-Tenant Reports

FlowAdmin Portal → GET /api/reports/transactions
FlawReport query uses merchant_id from session, but falls back to returning all records when merchant_id is null (system admin context). If T-03 (IDOR) is exploited, attacker can null out this field.
WhoAttacker who has exploited T-03 to elevate privileges
ImpactFull platform transaction data disclosure across all merchants

Risk Register & Mitigations

🎯 Activity 2: Critical Assessment

Sorted by risk level, then business impact. Each High/Critical finding includes three remediation horizons.

Critical

Critical

T-01 — JWT Algorithm Confusion (alg:none)

Business Impact: Full authentication bypass — any attacker can impersonate any user or admin without credentials.
NowEnforce RS256 with explicit algorithm whitelist in JWT library; reject tokens with alg:none or any symmetric algorithm
SoonAdd automated test in CI asserting that alg:none tokens return 401; rotate all existing signing keys
LaterShort-lived access tokens (15 min) + refresh token rotation; implement token binding
Critical

T-02 — SQL Injection on Transaction Queries

Business Impact: PCI DSS breach, full database exfiltration, regulatory fines up to 4% global revenue (GDPR), card brand penalties.
NowAudit all DB query construction; convert raw string concatenation to parameterized queries / prepared statements immediately
SoonIntegrate Semgrep with SQL injection rules into CI; block merges that introduce raw SQL patterns
LaterMigrate query layer to ORM (Sequelize / TypeORM) that prevents raw SQL by default; WAF rules for SQLi patterns
Critical

T-03 — Admin Portal IDOR

Business Impact: Any low-privilege admin can access all merchant data — full platform data disclosure, regulatory exposure.
NowAdd server-side merchant ownership check to every /admin/merchants/{id}/* endpoint; return 403 if requester's merchant_id doesn't match
SoonCentralize authorization to a single middleware; write integration tests covering cross-merchant access attempts
LaterAttribute-based access control (ABAC) with explicit policy definitions per endpoint; quarterly access review
